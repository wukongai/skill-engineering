# 能力探索与产物判断

这一步发生在创建 Skill 之前。目标不是把用户的第一句话包装成 Skill，而是理解真实任务并选择维护成本最低的正确产物。

## 两种入口不要混淆

### 新需求

顺序是：自查现有能力 → 与用户逐步澄清 → 比较方案 → 判断是否需要 Skill。

此时不得先创建 Product、版本、Backlog 或脚手架。

### 恢复已有 Skill 项目

已经确认存在并进入维护的项目，可以先读取 Product、Architecture、当前版本、Sprint 和 Task 恢复上下文。

## 先自查

提问用户之前，自动检查当前可见范围内：

- 已有 Skill 及触发边界；
- 项目规则和文档；
- scripts、CLI 和确定性工具；
- Plugin/MCP/runtime 能力；
- Profile、安装入口和重复暴露；
- 已有失败记录、用例或相似实现。

自查能回答的问题不要再让用户手工盘点。发现可能重叠时，优先讨论复用、扩展或替换。

## 再逐步提问

一次只问一个会改变产物选择的问题。优先顺序：

1. 最近一次真实任务是什么，期望结果是什么？
2. 它是一次性任务，还是未来会重复发生？
3. 哪部分必须由 AI 判断、解释或编排？
4. 给出一个应该触发和一个不应该触发的相似请求。
5. 用户实际会提供什么输入，必须拿到什么输出？
6. 已有能力为什么不能复用或扩展？
7. 最不能再次发生的失败或风险是什么？

已有上下文足以回答时跳过对应问题。简单需求可以用一句设计确认，不要为了流程强迫用户回答完整问卷。

## 比较 2–3 个方案

信息足够后，使用普通用户能理解的语言比较最相关的 2–3 个方向，例如：

- 不新增产物，直接完成当前任务；
- 扩展一个已有 Skill；
- 新建 Skill；
- 下沉为 Script；
- 使用 Plugin/runtime；
- 只增加项目规则或 Profile 暴露。

推荐一个方案，并说明复用价值、维护成本、触发风险、运行依赖和安全边界。不要只列菜单让用户自己做工程判断。

## 判断门

只有满足以下条件才进入 Skill 创建：

- 任务会重复，或具有稳定的团队/商业复用价值；
- 核心需要 Agent 判断、解释、路由或编排；
- 应触发和不应触发的边界可以分开；
- 已有 Skill 不能在不膨胀职责的情况下覆盖；
- runtime 和确定性部分已经正确下沉到 Plugin/Script；
- 用户认可推荐方向和最小能力边界。

判断结果可以是 `needs_discovery`、不新增、扩展已有、Script、Plugin/runtime、项目规则、Profile 或新 Skill。信息不足时必须保持 `needs_discovery`，不得默认创建 Skill。

## 进入后续阶段

确定创建或扩展 Skill 后：

1. 选择 atomic/orchestrator/router/adapter/composite；
2. 选择轻量、团队/生产或商业/工业治理等级；
3. 定义输入、输出、停止点、安全和失败路径；
4. 只有复杂、多版本或商业 Skill 才进入 Product、版本、Backlog、Spec/Plan、ADR 和 Sprint 治理。
