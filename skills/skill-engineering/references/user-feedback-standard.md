# 用户反馈标准

Skill Engineering 面向只想完成任务的普通用户。工具记录用于审计和恢复,不能直接充当用户反馈。

项目内所有确定性反馈格式以 `docs/guides/skill-interaction-design.md` 为实现规范;本文件负责 Skill 对话层的摘要约束。

## 默认首屏

按以下顺序回答,没有内容的项可以省略:

1. **结果**:成功、尚未完成或被安全阻止。
2. **影响**:用户现在能做什么;真实项目、全局环境或外部系统是否被修改。
3. **下一步**:继续自动执行,或只提出一个用户可理解的决定。

默认不展示命令、原始 JSON、内部 ID、hash/fingerprint、状态枚举、CandidateJob、Shadow、Canary、ReleaseRecord 等实现术语。用户明确要求排障、复现、审计或技术详情时,再增加“技术详情”,且先给普通摘要。

## 审批表达

审批问题必须包含:

- 将执行的实际动作;
- 动作发生的位置或范围;
- 是否影响真实项目或外部系统;
- 是否可以撤销。

不要说:

> 请批准 Canary 应用 release-plan-xxx。

应该说:

> 测试版本已经准备好。下一步只会把它临时接入隔离测试项目,不会修改真实项目,验证后可以完整移除。是否继续?

## 成功与失败

- 完全成功:直接说明交付结果和用户现在可以做什么。
- 部分成功:明确说“整体尚未完成”,分别说明已完成和未完成部分。
- 安全阻止:先说明没有发生哪些副作用,再说缺少什么和如何继续。
- 自动恢复:说明内容是否恢复、记录是否完整;两者任一失败都不能声称整体通过。

不要说:

> Canary 闭环通过;维护 undo 的内容恢复通过,但操作记录终态验收失败。

应该说:

> 测试安装和移除已经完成,真实项目没有变化。最后清理时,文件已恢复,但系统没有正确记下这次恢复,所以整项测试还不能算通过。下一步需要先修复这处记账问题。

## Handoff 摘要

恢复任务时只展示:

- 用户原本想完成什么;
- 已经得到的可感知结果;
- 当前是否安全;
- 唯一下一步。

内部记录 ID 和复现命令仅在技术详情中提供,不得要求用户重述已有信息。
