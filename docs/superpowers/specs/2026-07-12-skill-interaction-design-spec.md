# Skill Engineering 交互体验设计 Spec

## 背景

真实端到端测试表明，Skill Engineering 的工程门禁可以正确执行，但默认反馈仍会泄露计划 ID、记录 ID、指纹、发布通道和状态枚举。用户无法直接判断任务是否完成、真实项目是否安全以及下一步是什么。

本次改造把产品交互从“展示内部工作流”升级为“管理用户委托”。

## 目标

1. 所有默认人类可读输出统一采用“结果、影响、下一步”。
2. 写入、安装和发布审批描述实际动作、范围、影响和撤销能力。
3. 部分成功明确为整体未完成，不制造假完成。
4. Journey 跨窗口恢复不暴露内部 stage/status，且不要求用户重述。
5. JSON 和持久化 schema 保持兼容，继续服务自动化、审计和恢复。
6. Skill Engineering 的自然语言规则与 Python 核心的确定性输出遵守同一规范。

## 非目标

- 不构建图形界面。
- 不移除任何 JSON 字段或机器记录。
- 不改变评测、发布和维护的安全门禁。
- 不自动执行 Global/Profile 安装或外部平台同步。

## 用户模式

### 普通模式

默认模式。输出结果、影响、下一步和必要的单个决定，不显示内部实现词。

### 解释模式

用户追问“为什么”时，展示证据、替代方案、未知项和结论边界。

### 技术模式

用户显式使用 `--json` 或要求技术详情时，返回完整机器记录。现有 JSON 行为不变。

## 统一消息契约

新增 `UserFeedback`：

- `status`：completed、incomplete、blocked、awaiting-approval、in-progress；
- `result`：一句自然语言结论；
- `impact`：零到三条用户影响；
- `next_action`：唯一下一步；
- `decision`：仅在需要用户决定时出现；
- `technical_details`：默认不渲染。

渲染时不显示内部状态值，只使用它选择自然语言表达。

## 关键流程要求

### Journey

恢复摘要必须包含目标、最近可感知结果、当前安全状态和下一步。不得直接输出 `stage(status)`。

### 创建和改进计划

默认展示要创建或修改的文件类别、真实目标、是否已写入和下一步。统一 diff、计划 ID 和复杂度明细留在 JSON 或技术详情。

### 评测

先说候选是否可以接受，再分开说明真实行为覆盖、风险用例和结论边界。Suite ID 与 subject fingerprint 不在默认首屏。

### 发布

发布预览说明接入范围、真实项目影响和撤销能力；发布后说明验证是否通过；回滚后说明入口是否已经移除。

### 失败

如果内容恢复但记录失败，必须输出“整体尚未完成”，同时说明真实文件已经恢复和记录仍有问题。

## 兼容性

- `--json` 输出字段和 schema 不变。
- 已保存 Journey、Plan 和 Record 无需迁移。
- 默认文本变化属于产品界面升级，不影响自动化消费者。

## 验收标准

- 默认 Journey、Release Plan、Release、Rollback 和 Evolution 输出不出现内部 ID、指纹或裸状态枚举。
- 审批文案包含动作、范围、影响和可撤销性。
- 部分失败不会出现“全部通过”或“完整闭环”。
- 现有测试全部通过。
- production Doctor 仍为零 FAIL。
- 至少两个独立真实对话前向测试能让普通用户直接回答“是否完成、影响了什么、下一步是什么”。
