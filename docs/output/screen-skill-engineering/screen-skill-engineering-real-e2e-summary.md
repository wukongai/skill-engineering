# Skill Engineering 真实用户实操：可展示摘要

## 一句话叙事

Skill Engineering 把“我想做一个 Skill”变成一条可审阅、可确认、可回归的工程流程：先判断是否值得做成 Skill，再逐步收敛边界，确认后写入；真实试用出现问题时，先定位根因、生成独立候选、展示 diff，确认后再应用并校验。

## 可展示的流程

```text
描述重复工作
  ↓
只问会改变方案的问题
  ↓
比较不新增产物、项目规则和独立 Skill
  ↓
展示职责边界与文件计划
  ↓ 用户确认
创建最小可用 Skill
  ↓
真实试用
  ↓
按失败层级定位问题
  ↓
独立候选 → diff 审阅 → 用户确认 → 应用 → 结构校验
```

## 三个营销重点

1. **确认前不写入**：用户先看到结果、影响范围和将创建的文件，再决定是否落地。
2. **修改不堆补丁**：失败后判断属于触发边界、处理规则、输出格式还是验证层，先做候选再应用。
3. **结果可验证**：创建和维护都有结构校验；回归 fixture 固化“不得把原文没有说过的内容写进去”。

## 真实边界

- 本次真实实操验证了“安装 → 发现 → 确认 → 创建 → 试用反馈 → 候选修改 → 应用 → 校验”的完整流程。
