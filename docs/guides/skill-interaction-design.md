# Skill 交互设计规范

Skill 的交互不是“给 CLI 套一个聊天框”，而是管理用户与 Agent 之间的委托关系。

## 核心区别

传统 App 以“用户操作系统”为中心，提前提供页面、控件和固定路径。Skill 以“用户委托目标”为中心，需要在开放语言中完成目标理解、必要澄清、行动授权、结果验证和失败恢复。

因此，Skill 的默认交互目标不是展示系统做了多少工作，而是让用户始终知道：

1. 结果是什么；
2. 对自己或真实项目有什么影响；
3. 下一步是否需要做决定。

## 三层信息结构

### 用户结果层

默认展示结果、影响和下一步。避免命令、原始 JSON、内部 ID、hash、fingerprint、状态枚举和实现层名称。

### 过程解释层

用户询问原因时，展示推荐依据、已检查范围、未知项和停止原因。

### 技术证据层

用户明确要求排障、复现或审计时，展示机器记录、命令、路径、指纹和日志。技术证据必须保留，但不能直接充当用户结论。

## 交互阶段

### 理解目标

接受自然语言目标，自动检查已有能力。不要要求用户先理解 Skill、Plugin、Script、Profile、Contract 或内部阶段。

### 必要澄清

只询问会改变结果、作用范围、安全边界或不可逆副作用的问题。通常一次问一个，最多同时三个。

### 推荐方案

说明建议做什么、为什么、为什么不选主要替代方案，以及当前仍不知道什么。

### 行动预览

写入前必须说明实际动作、发生位置、真实影响、不会发生什么、是否可以撤销。

### 进度反馈

只在可感知里程碑、方向变化、风险变化或需要用户决定时更新。不要逐条转播命令和工具日志。

### 结果交付

说明用户现在能做什么，哪些真实系统被修改或没有修改，验证覆盖到哪里。结构健康与真实任务效用必须分开表达。

### 部分失败

任何必要环节失败时，明确写“整体尚未完成”，分别说明已经完成、仍未完成、当前是否安全和唯一下一步。

### 中断恢复

跨窗口恢复只展示用户原始目标、可感知结果、当前安全状态和唯一下一步，不要求用户重述已经提供的信息。

## 审批规则

用户批准的是实际行动，不是内部对象。审批问题至少包含：

- 要执行什么；
- 在哪里执行；
- 是否影响真实项目或外部系统；
- 是否可撤销。

错误示例：

> 是否批准 Canary Release Plan？

正确示例：

> 测试版本已经准备好。下一步只会把它临时接入隔离测试项目，不会修改真实项目，验证后可以完整移除。是否继续？

## 事实等级

- 已验证：存在真实运行结果和可重算证据；
- 已声明：契约存在，但没有真实行为结果；
- 推断：基于当前证据的判断；
- 未知：证据不足，不能下结论；
- 已阻止：因安全或授权边界没有继续。

不得把静态结构分数写成真实效果，也不得把工具返回成功写成用户目标已经完成。

## 评测要求

至少覆盖成功、缺输入、高风险、未知事实、部分恢复失败和跨窗口恢复。除任务通过率外，还需观察：

- 用户提问次数；
- 首次有用建议所需轮次；
- 默认反馈中的内部术语数量；
- 用户能否理解自己授权的动作；
- 假完成率；
- 恢复和撤销是否只影响本次拥有的动作。
