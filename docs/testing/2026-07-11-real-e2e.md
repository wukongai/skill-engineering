## 真实需求

被测能力为 `article-release-review`：仅在用户明确要求文章发布前事实、证据或风险审查时触发；普通润色、标题和排版不触发。它必须：

- 对数字、引语和产品声明检查来源；
- 在来源缺失或内容不支持声明时返回 `blocked`，不得编造来源；
- 在凭证、隐私或未确认发布等高风险边界返回 `blocked`；
- 只给出审查结论，不自行发布。

测试使用独立临时 root、候选工作区和 consumer project，不接触用户的真实 Skill、项目入口或远端服务。

## 测试发现并修复的问题

### 1. 生产级 orchestrator 脚手架缺少完整评测契约

创建器未生成 baseline、holdout、negative-transfer、类型检查与显式状态契约，导致 production Doctor 报 `EVAL103/EVAL104`。修复后，生产级复杂 Skill 默认生成这些工程证据入口和 holdout fixture。新建但尚未评测的 Skill 仍会保留 `EVAL107`，这是防止占位契约冒充真实行为证据的预期门禁；完成行为评测并附加有效报告后才降为 0 fail。

### 2. 行为报告声明格式与 Doctor 读取协议不一致

创建器输出字符串路径，Doctor 只认证 `behavioral_results.report` 映射，导致真实 `accept` 报告仍被降级为 `evidence-declared`。现在脚手架直接生成可验证报告协议，并有组合回归测试。

### 3. A3 数据和结果证据存在完整性缺口

- 数据集生成器写出 `high-risk`，评测 schema 只接受 `high_risk`，真实高风险候选无法提交；现已统一枚举并加入端到端 schema 回归。
- `submit-results` 原先未绑定结果 artifact 与 Proposal baseline/候选源码指纹，存在跨候选冒用结果的可能；现在 baseline 与 candidate 指纹必须分别精确匹配，否则拒绝提交。

## 能力边界

这次测试证明：当前实现能够在一份有哈希的确定性 suite 和结果 artifact 上执行完整工程闭环，并正确处理留出集、高风险门禁、负迁移、版本、发布和回滚。

它不证明跨模型、跨长期真实业务的普遍效用。行为输出由本次宿主测试 harness 生成，不是独立 LLM reviewer 的语义评审；因此效用结论保持为限定范围内的 `behavioral-evidence`。下一版本接入 LLM 时，应在不改变现有确定性门禁的前提下增加独立 rollout 与 reviewer 证据。
