# Skill Engineering 产品定位

## 一句话价值

帮助用户快速生成从第一版开始就符合工程规范的 Agent Skill，并让它在每次维护、扩展和发布后依然清晰、可靠、可演进。

## 核心问题

快速生成必须是第一等能力，但“生成出来”不能以牺牲架构为代价。Skill 从第一次生成开始就应具备清楚边界、合理结构、安全控制和验证入口；在多次修改、多人协作、模型切换和商业交付后，这些约束仍然必须成立。

Skill Engineering 的核心竞争力是“架构优先的快速生成 + 全生命周期抗腐化”，而不是只生成更多 Prompt 文本，也不是等 Skill 变乱后再补治理。

## 目标用户

- 不熟悉 Skill 工程规范、只会描述业务目标的普通用户；
- 维护多个个人或团队 Skill 的开发者；
- 构建复杂编排、商业交付或工业流程 Skill 的团队；
- 需要评测、版本、审批、发布和回滚证据的组织。

## 产品层次

### 简单 Skill

帮助用户判断是否值得创建，并快速生成最小但完整、清楚、安全、可以继续维护的 Skill。

### 团队/生产 Skill

增加 contract、回归用例、行为证据、维护记录和发布门禁。

### 商业/工业级 Skill

增加产品边界、版本、Backlog、Spec/Plan、ADR、审计、安全策略、独立评审和发布/回滚证据。

## 核心体验

用户只需要描述目标。Skill Guide 自动完成盘点、提问、架构选择、完整候选生成、计划、检查和验证；只有真正会改变文件、安装范围或外部状态时才要求确认。

默认输出只回答：结果是什么、实际影响是什么、下一步是什么。

## 用户需求流程

```text
用户描述需求
  -> 自查已有 Skill / Script / Plugin / 项目规则
  -> 一次一个问题澄清真实任务
  -> 比较 2-3 个产物方案
  -> 决定不新增 / 扩展已有 / Script / Plugin / Skill
  -> 确认需要 Skill 后选择架构和治理等级
  -> 复杂或商业 Skill 才进入 Product / 版本 / Backlog / Spec / Plan
```

“恢复一个已经存在的 Skill 产品”是另一种入口，可以先读取 Product、版本和当前 Sprint；不能把这个恢复流程套在尚未判断是否需要 Skill 的新需求上。

## 开源策略

v0.1 采用 local-first、provider-neutral 的 MIT 开源核心。训练营、案例、团队协作、托管评测和企业治理可以在开源核心之上形成服务，但不得通过封闭格式锁定用户。

用户依赖 Skill Engineering 的原因应是它持续提供架构保护、行为证据、演进历史和安全发布能力。
