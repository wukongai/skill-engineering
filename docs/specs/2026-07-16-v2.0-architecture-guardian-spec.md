# Skill Engineering v2.0 Architecture Guardian Spec

## 背景

0.1.x/1.0 负责检查单个 Skill 的结构、安全和行为证据，但重复维护多个 Skill 后仍会出现更高层的架构问题：职责重复、路由冲突、依赖不透明、入口过厚、治理等级与实际风险不匹配。2.0 要把这些问题提升为机器可读的架构事实和可审计变更计划。

## 核心目标

Architecture Guardian 必须能回答：

- 这个 Skill 由哪些组件组成，各组件承担什么角色？
- 执行拓扑和副作用边界是什么？
- 它依赖哪些 Skill、Script、Plugin、Provider 或外部资源？
- 这次修改是否引入重复职责、route collision、context budget 超限或治理降级？
- 如果架构不合适，系统能否给出拆分、迁移、压缩或废弃候选？

## Blueprint 三轴

1. **Component role**：atomic、router、orchestrator、adapter、reference、script、test 等角色。
2. **Execution topology**：入口、阶段、delegate、状态、外部副作用和回滚边界。
3. **Governance level**：personal、team、production、commercial 及其证据要求。

## 范围

- Blueprint/IR 的版本化 schema 和校验；
- 从现有 `SKILL.md`、references、scripts、contract 生成只读 Blueprint；
- 依赖图、重复职责、冲突触发和 route collision 检查；
- context budget、入口厚度、状态/副作用与治理等级适配度检查；
- baseline/candidate semantic diff；
- 把 guardian finding 转成 improve/create 的 preview plan；
- 1.x contract/Doctor/evaluate/release evidence 的兼容读取。

## 非目标

- 不替代 Skill Guide 的自然语言路由；
- 不自动修改或自动拆分生产 Skill；
- 不负责 Agent Skill Hub 的 registry/profile/Global/多项目分发；
- 不用一个总分替代人工 review、行为评测或安全门禁；
- 不在第一阶段引入云端数据库、强制 LLM 或新 provider。

## 兼容与安全

- Blueprint 先只读生成，任何写入必须经过同一 immutable plan；
- 旧 Skill 没有 Blueprint 时生成 `legacy/unknown`，不能静默判定为失败；
- 所有建议必须带 finding、证据路径、预期复杂度变化、回归要求和撤销入口；
- schema 破坏性变化升级 Blueprint major version，并提供迁移工具。

## 验收标准

1. 给定一个 1.x Skill，可生成稳定、可校验、无凭证的 Blueprint JSON。
2. Blueprint 明确组件角色、执行拓扑、治理等级和未知字段。
3. 对 baseline/candidate 能报告新增、删除、变化的组件和边界，而不是只比较文本行。
4. 能发现重复职责、route collision、未声明依赖和明显 context budget 超限。
5. guardian 只生成 preview plan，不直接覆盖目标。
6. 旧 Doctor、evaluate、improve 和 release 流程保持兼容。
7. 至少有 schema、negative、high-risk、migration 和 rollback regression cases。
