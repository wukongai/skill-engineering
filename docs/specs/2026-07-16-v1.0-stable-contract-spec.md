# Skill Engineering v1.0 Stable Lifecycle Contract Spec

## 背景

0.1.x 已经证明了本地创建、Doctor、评测、维护、演进和发布治理闭环，但 CLI、JSON、版本源、安装 smoke 和迁移策略仍未冻结。1.0 的任务不是继续堆功能，而是把已验证能力变成可依赖的稳定契约。

## 目标

1. 用户能从 README 安装并完成 create → doctor → evaluate → improve → release 的本地闭环。
2. 现有 CLI、JSON、contract、计划和记录格式有明确 schema/version 与兼容策略。
3. 发布结果有可复现证据，不把结构分数或单次静态检查当作真实效用。
4. 1.x 的新增能力可以兼容演进，破坏性变化明确留给 2.0。

## 范围

- 稳定 CLI/API/JSON schema 与错误码；
- package/wheel、安装 smoke、Python 支持矩阵；
- release evidence、迁移、回滚和废弃策略；
- 1.0 用户文档、贡献流程、安全和支持边界；
- 与现有 Skill Engineering、Doctor、evaluate、maintenance、evolution 对齐。

## 非目标

- Blueprint/IR 和 Architecture Guardian 主架构；
- 云端协作、RBAC、托管评测、自动 Global 发布；
- 新增 LLM provider 或把静态分数改成效用分。

## 验收标准

1. `pyproject.toml`、运行时版本、README、CHANGELOG、发布日志和 Roadmap 一致。
2. CLI 主要命令有稳定的 help、退出码、JSON schema version 和错误边界。
3. 旧 0.1.x 计划/contract 至少能被读取或给出明确迁移错误。
4. wheel 构建、干净环境安装和最小 smoke 流程可重复。
5. production release evidence 包含 baseline、holdout、high-risk、negative-transfer 和独立评审入口。
6. 每个发布候选都有测试、凭证 lint、Doctor、diff check 和回滚记录。
7. 公开文档不把 Unreleased 或 evidence-declared 说成 Stable/已验证效用。

## 完成定义

Spec、Plan、实现、测试证据、Changelog、版本源、发布日志和用户文档一致，并获得独立评审；没有用户授权时不执行公开发布。

## 验证映射

- 版本事实一致性：`scripts/check-release.py`、`tests/test_v1_release.py`；
- 安装、四个 Use Case 与完整门禁：`docs/testing/2026-07-18-v1-use-cases.md`。
