# Skill Engineering v0.1 Public Beta Spec

## 背景

Skill Engineering 已经具备能力判断、Skill 创建、Doctor、行为评测、持续维护、自进化和安全发布的主体能力，但仓库自身缺少统一的产品版本、Backlog、Sprint、ADR 和日常开发记录；创建链也没有达到维护链已有的“同一计划、写后验证、失败恢复”标准。

如果这些缺口继续存在，Skill Engineering 会出现它试图替用户解决的同一种问题：功能持续增加，但产品边界、架构事实源和版本验收逐渐漂移。

## 产品定位

Skill Engineering 是 Agent Skill 的工程化创建、维护和演进系统。它帮助普通用户判断是否需要 Skill，并让简单 Skill 到商业/工业级 Skill 都能在反复修改后保持：

- 从第一版开始即可使用且符合工程规范；
- 清晰的触发与能力边界；
- 与复杂度匹配的架构；
- 可执行的安全和审批边界；
- 可验证的行为证据；
- 可审计、可撤销的维护与发布过程；
- 对普通用户友好的对话式交互。

核心卖点是“快速生成、从第一次生成开始工程化，并在修改很多次后仍然保持同一套架构与质量约束”。快速生成与抗架构腐化是同一个生命周期承诺，不是先生成、变乱后再治理的两个阶段。

## 本版本目标

v0.1 Public Beta 必须可以向小范围用户交付以下闭环：

1. 先自查已有 Skill、Script、Plugin/runtime、项目规则和相似实现，再通过一次一个问题澄清真实任务；信息不足保持 `needs_discovery`。
2. 信息足够后比较 2-3 个相关产物方案，再判断是否值得创建 Skill，或应扩展已有能力、改用 Script、Plugin/runtime、项目规则、Profile 或不新增产物。
3. 只有确认需要 Skill 后，才按 atomic、orchestrator、router、adapter、composite 选择最小合理架构和治理等级。
4. 根据已确认架构快速生成最小但完整的可用候选，而不是只创建空目录或无边界 Prompt；复杂内容由宿主 Agent 在隔离工作区生成。
5. 为复杂、有副作用或生产 Skill 生成 contract、成功/失败/高风险用例和必要的 holdout 入口。
6. 在创建和维护时默认先预览，不直接覆盖目标。
7. 创建必须引用已经预览的同一份未漂移计划；写入后自动做结构验证，结构失败时清理整个新目标。
8. production 创建后明确区分“结构可继续开发”和“已有真实行为证据、可生产发布”。
9. 修改已有 Skill 时要求失败模式、根因层级、预期行为和回归证据，并检查复杂度、重复指令和入口膨胀。
10. 支持确定性 baseline/holdout/high-risk/negative-transfer 评测、自进化候选、不可变版本和安全回滚。
11. 默认交互只说明结果、真实影响和唯一下一步，不要求普通用户理解内部 ID、状态和哈希。
12. Skill Engineering 使用自己的规范管理自己。

## 正确流程顺序

对于新需求：Discover/Brainstorm → Artifact Decision → Skill Architecture → Governance Level → Create/Maintain。

Product、版本、Backlog、Sprint、Spec/Plan 只在已经决定创建/扩展复杂或商业 Skill 后进入。对于恢复已有 Skill 产品，可以从 Product 和当前版本进入。

## 自举工程体系

仓库必须建立以下单一事实源：

| 工件 | 职责 |
|---|---|
| `docs/PRODUCT.md` | 产品定位、用户、价值和版本边界。 |
| `docs/constitution.md` | 长期不可随意漂移的工程原则。 |
| `docs/architecture.md` | 当前实现架构和模块边界。 |
| `docs/ROADMAP.md` | 版本级演进方向。 |
| `docs/BACKLOG.md` | 尚未承诺进入当前版本的候选需求。 |
| `docs/TASK.md` | 当前版本正在执行和待验收的任务。 |
| `docs/specs/` | 功能为什么做、必须满足什么。 |
| `docs/plans/` | 如何实施、如何验证。 |
| `docs/adr/` | 跨版本架构决策和取舍。 |
| `docs/sprints/` | 一个交付周期的目标、范围和退出条件。 |
| `docs/logs/daily/` | 当天事实、验证和未解决问题，不代替长期规范。 |
| `CHANGELOG.md` | 已发布版本的用户可感知变化。 |

Daily Log 和事故信息不得继续堆入根 `SKILL.md`。只有稳定、跨版本仍成立的规则才能进入 Skill 入口或正式规范。

## 分级治理

### 简单 Skill

保持轻量。默认只需要 `SKILL.md`；只有出现稳定失败、复杂状态或副作用时才升级结构。

### 团队/生产 Skill

需要 contract、测试用例、行为证据入口、维护记录和版本边界。

### 商业/工业级 Skill

在生产级基础上，要求产品边界、Roadmap/Backlog、Spec/Plan、ADR、版本与变更记录、安全审查、独立评审和发布/回滚证据。

不得把商业治理模板强塞给一次性或简单 atomic Skill。

## 创建安全契约

### Preview

- `skill-engineering create ...` 只生成和保存计划。
- Preview 不写目标目录。
- 计划包含目标指纹和自身哈希。

### Apply

- 只能使用 `skill-engineering create --plan <ID> --apply`。
- 不允许在 `--apply` 时重新解释用户参数并生成另一份计划。
- 目标、计划内容或 schema 漂移时拒绝写入。

### Postflight

- 写入后运行 lint 和 team Doctor，作为结构门禁。
- 结构门禁失败时删除本次新建目标并保存失败状态。
- production 额外运行 production Doctor 形成发布就绪报告。
- 缺真实行为证据可以让 production readiness 保持 incomplete，但不能冒充发布完成，也不应清理结构正确的脚手架。

## Skill Guide 引导契约

当用户提出创建、修改、重构、评测、版本升级或商业化 Skill 时，Skill Guide 必须自动进入相应工程流程。

对于复杂或商业 Skill，Skill Guide 应先检查是否存在产品边界、架构、Backlog、版本、Spec/Plan 和 ADR；缺失时生成最小治理计划，而不是把管理信息写入根 SKILL.md。

## 非目标

以下内容进入后续 Backlog，不阻塞 v0.1 Public Beta：

- 完整的跨平台 Skill IR 和多目标 compiler；
- 内置模型 provider runner；
- HTML Review Studio；
- 云端协作、账号、计费和企业 RBAC；
- 原生客户端 telemetry；
- 自动公开发布、自动 push 或全局安装。

## 开源发布要求

- README 清楚说明 Public Beta 能力和限制。
- MIT License 保持有效。
- 提供贡献指南、安全报告入口和文档索引。
- GitHub CI 至少运行 pytest、Ruff、凭证 lint 和 production Doctor。
- 版本号、包版本和 Changelog 一致。
- 不包含 token、cookie、私钥、私有客户内容或本机绝对路径。

## 验收标准

1. 缺关键答案时 `decide` 返回 `needs_discovery`,一次只向用户显示一个问题,不默认创建 Skill。
2. 信息足够时给出明确产物判断和相关替代方案。
3. 创建 Preview 不写文件。
4. 直接 `create --apply` 被拒绝。
5. 同一计划 Apply 成功并保存 postflight 结果。
6. 计划或目标漂移会阻断 Apply。
7. 无效创建内容触发 postflight 清理，不留下半成品目录。
8. production 脚手架显示结构结果与生产证据缺口，不制造假完成。
9. Skill Guide 明确覆盖创建、修改、防架构腐化、检查、自进化和商业治理触发。
10. README、Product、Spec 和 Skill 入口把快速生成与全生命周期架构保护表达为同一个产品承诺。
11. 自举文档体系完整且互相链接。
12. 全部 pytest、Ruff、production Doctor、凭证 lint 和 diff check 通过。
13. 至少一次隔离 CLI E2E 验证 preview -> same-plan apply -> postflight。
