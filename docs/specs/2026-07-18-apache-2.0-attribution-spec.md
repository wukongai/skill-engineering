# Skill Engineering Apache-2.0、署名与品牌边界 Spec

状态：Accepted，作为 v1.0 Stable Lifecycle Contract 的发布前置项

## 背景

v1.0 候选阶段已经完成标准安装入口、canonical identity 和 MIT 版权人“艾笑”的对齐。随后产品目标进一步收敛：普通用户应当能够自由使用和商业使用，开发者再分发时应保留项目源头、许可证与署名信息，并为未来商业服务和营销入口保留清晰的品牌边界。MIT 可以要求保留版权和许可文本，但缺少显式专利授权、修改声明与 `NOTICE` 机制，因此不再是当前目标的最佳匹配。

## 目标

1. 将仓库当前版本的许可证从 MIT 迁移为 Apache License 2.0，版权人为“艾笑”。
2. 通过 `LICENSE`、`NOTICE` 和包元数据建立可机器识别、可追溯的来源链。
3. 通过 `CITATION.cff` 提供推荐引用方式，通过 `TRADEMARKS.md` 分离代码许可与品牌授权。
4. 保持普通用户、个人用户和商业用户均可使用、修改与分发的开源边界。
5. 让 README、产品文档、贡献指南、安装指南、版本与发布材料对同一政策给出一致说明。

## 决策范围

### 许可证与署名

- 当前版本使用 `Apache-2.0`，版权人为“艾笑”。
- 默认覆盖仓库原创的源代码、Python 包、CLI、Agent Skill 指令、references、schemas、tests、examples 和文档。
- 再分发者必须遵循 Apache-2.0：提供许可证副本，保留适用的版权、专利、商标和署名通知；修改文件时给出显著修改声明；包含衍生作品时以许可证允许的方式保留 `NOTICE` 中的适用署名。
- `NOTICE` 仅记录信息性署名和 canonical source，不增加 Apache-2.0 之外的使用限制。
- `CITATION.cff` 是推荐引用方式，不构成额外许可条件。
- Apache-2.0 不强制公开修改后的源代码，也不保证在产品界面中展示营销署名。

### 内容、贡献与品牌

- 第三方代码、数据、商标、截图或引用继续遵循各自条款；新增捆绑内容必须记录来源和兼容性。
- 用户自己的 prompt、私有数据、生成的 Skill 和运行产物不由本项目主张所有权，用户仍需确认输入、模型和第三方素材的权利。
- 贡献默认按 Apache-2.0 进入项目；贡献者不得提交无权授权的内容。
- Apache-2.0 不授予使用项目名称、Logo 或其他品牌标识来暗示官方、背书或合作关系的权利。描述兼容性、来源和事实性引用不受影响。

### 安装与运营

- Python CLI 与 Agent Skill 仍是两个独立交付物，安装一个不会自动安装另一个。
- Agent Skill 继续以标准 `skills` CLI 为普通用户入口，canonical 名称保持 `skill-engineering`。
- 对外可表述为“Apache-2.0 开源核心、允许商业使用、可自托管、保留来源与通知义务”；不得表述为“必须在产品界面宣传艾笑”或“所有衍生作品必须开源”。
- GitHub 发现与营销入口依靠清晰 README、仓库 topics/description、release 质量、`CITATION.cff` 和 canonical source，而不依赖附加的非标准许可证限制。

## 非目标

- 不创建自定义“必须公开署名”条款，不把本项目伪装成 OSI 标准许可证之外的限制性开源。
- 不要求衍生作品公开源代码，不建立 copyleft。
- 不在本 Spec 中执行 tag、GitHub Release、PyPI 发布、push 或 Global/Profile 安装。
- 不建立商标注册、云服务条款、SLA、数据处理协议或商业价格表。

## 验收标准

1. 根目录 `LICENSE` 是未改写附录占位符的官方 Apache License 2.0 全文，`NOTICE` 的版权人与 `pyproject.toml` 的作者均为“艾笑”，包元数据使用 `Apache-2.0` SPDX 标识。
2. 根目录存在 `NOTICE`、`CITATION.cff` 和 `TRADEMARKS.md`，内容不引入与 Apache-2.0 冲突的附加限制。
3. README、PRODUCT、CONTRIBUTING、版权安装指南、TASK、VERSIONING、CHANGELOG、Sprint 和发布证据不再把当前版本描述为 MIT。
4. ADR 0005 被明确标记为由新 ADR 取代；历史版本的 MIT 事实可以保留，但不得与当前政策混淆。
5. 自动化测试验证许可证文本、包元数据、版权人、NOTICE 和 citation 的关键字段。
6. pytest、Ruff、官方 Skill validation、production Doctor、credential lint、文档链接检查和 `git diff --check` 全部通过。
7. PM 独立复核没有阻断级问题，OB task 与仓库 TASK/日志完成对齐。

## 风险

- Apache-2.0 的署名义务可能被误解为强制前台宣传；文档必须区分“保留法律通知”和“营销展示”。
- 当前 v1.0 尚未正式 tag，迁移可以在首个稳定版前完成；一旦已有外部分发副本，旧副本仍按其获得时的 MIT 条款使用。
- 品牌政策过度扩张会削弱开源采用；`TRADEMARKS.md` 只阻止混淆官方来源，不限制事实性引用和兼容性说明。
